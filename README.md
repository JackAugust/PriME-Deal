# PriME‑Deal Reproducibility Guide

This document describes how to reproduce the on‑chain gas costs and
zero‑knowledge proof performance of PriME‑Deal as reported in the paper.
All experiments were performed on the following platform, but any
comparable Linux environment should yield similar results.

## Hardware & Software Requirements
- CPU: Intel Core i7‑11700 (2.50 GHz, 16 cores) or equivalent
- RAM: 32 GB
- OS: Ubuntu 20.04 LTS (tested) / 22.04 LTS (works)
- Node.js: v18 or v22 (tested with v22.22.3)
- Circom compiler: 2.1.8 (see installation below)
- snarkjs: 0.7.6 (installed locally via npm)
- npm / npx
- Hardhat 2.19.0 (installed locally)
- Python 3.6+ (for the cryptographic core; see Buyer/Seller benchmark README)

## 1. Smart Contract Gas Benchmark

This section measures the gas consumption of the PriME‑Deal smart contract
using a **real Groth16 proof** generated by the buyer circuit
(`buyer_nb500_tb5_m50.circom`).  The verifier contract is automatically
generated from the circuit’s verification key.

### 1.1 Setup
```bash
cd /path/to/priME-deal/src/contracts
npm init -y
npm install --save-dev hardhat@2.19.0 @nomiclabs/hardhat-waffle \
  @nomiclabs/hardhat-ethers ethers chai
```

### 1.2 Export the Verifier Contract
The circuit parameters (`n_b=500`, `t_b=5`, `m=50`) have already been
compiled and trusted setup completed.  Export the Solidity verifier:
```bash
npx snarkjs zkey export solidityverifier \
  ../../zkp_nb500_tb5_m50/buyer_nb500_tb5_m50_final.zkey \
  ./Verifier_zkp_nb500_tb5_m50.sol
```

### 1.3 Contract Files
Place the following two files in the current directory (`src/contracts`):
- `PriMEDeal.sol` – the main contract that inherits the verifier (see below)
- `Verifier_zkp_nb500_tb5_m50.sol` – the exported verifier

**`PriMEDeal.sol`** (already provided in the repository):
```solidity
// SPDX-License-Identifier: MIT
pragma solidity >=0.7.0 <0.9.0;

import {Groth16Verifier} from "./Verifier_zkp_nb500_tb5_m50.sol";

contract PriMEDeal is Groth16Verifier {
    struct Listing {
        address payable seller;
        uint256 v;
        uint256 d;
        bool active;
    }
    mapping(bytes32 => Listing) public listings;
    event Listed(bytes32 indexed sid, address seller);
    event Bought(bytes32 indexed sid, address buyer);

    function list(bytes32 sid) external payable {
        require(!listings[sid].active, "Already listed");
        listings[sid] = Listing({
            seller: payable(msg.sender),
            v: msg.value,
            d: msg.value,
            active: true
        });
        emit Listed(sid, msg.sender);
    }

    function buy(bytes32 sid,
                 uint[2] calldata a,
                 uint[2][2] calldata b,
                 uint[2] calldata c,
                 uint[14] calldata input) external payable {
        Listing storage l = listings[sid];
        require(l.active, "Not active");
        require(msg.value == l.v, "Incorrect payment");
        require(verifyProof(a, b, c, input), "Invalid proof");
        l.active = false;
        (bool ok, ) = l.seller.call{value: msg.value}("");
        require(ok, "Transfer failed");
        emit Bought(sid, msg.sender);
    }
}
```
*The `input` array size (14) must match the number of public signals in the
circuit.  If you use a different circuit configuration, adjust this length
and regenerate the verifier accordingly.*

### 1.5 Test Script
Create `test/gas.js` that reads the real proof and submits it to the contract.
**Note:** the paths to `proof.json` and `public.json` are relative to the
`test/` directory and point to the root `zkp_nb500_tb5_m50/` folder.

```javascript
const { expect } = require("chai");
const { ethers } = require("hardhat");
const fs = require("fs");
const path = require("path");

describe("PriMEDeal Gas (real ZKP)", function () {
    let contract, seller, buyer;
    const sid = ethers.utils.formatBytes32String("trade1");

    const proofPath = path.join(__dirname, "../../../zkp_nb500_tb5_m50/proof.json");
    const publicPath = path.join(__dirname, "../../../zkp_nb500_tb5_m50/public.json");
    const proof = JSON.parse(fs.readFileSync(proofPath, "utf8"));
    const pub = JSON.parse(fs.readFileSync(publicPath, "utf8"));

    const a = [proof.pi_a[0], proof.pi_a[1]];
    const b = [
        [proof.pi_b[0][0], proof.pi_b[0][1]],
        [proof.pi_b[1][0], proof.pi_b[1][1]],
    ];
    const c = [proof.pi_c[0], proof.pi_c[1]];

    if (pub.length !== 14) throw new Error(`Expected 14 public inputs, got ${pub.length}`);

    before(async () => {
        [seller, buyer] = await ethers.getSigners();
        const PriMEDeal = await ethers.getContractFactory("PriMEDeal");
        contract = await PriMEDeal.deploy();
        await contract.deployed();
    });

    it("list gas", async () => {
        const tx = await contract.connect(seller).list(sid, {
            value: ethers.utils.parseEther("0.1")
        });
        const receipt = await tx.wait();
        console.log(`list gas: ${receipt.gasUsed.toString()}`);
    });

    it("buy gas (real Groth16 proof)", async () => {
        const tx = await contract.connect(buyer).buy(sid, a, b, c, pub, {
            value: ethers.utils.parseEther("0.1")
        });
        const receipt = await tx.wait();
        console.log(`buy gas: ${receipt.gasUsed.toString()}`);
    });
});
```

## 2. Zero‑Knowledge Proof Performance Benchmark

This section reproduces the prover and verifier execution times for the
circuit `buyer_nb500_tb5_m50.circom`, which is representative of a
buyer with 50 attributes matching a policy of size 500 and threshold 5.

### 2.1 Prerequisites
- Circom compiler installed (see Section 1 of the original README)
- Node.js dependencies: `snarkjs@0.7.6`, `circomlib` (if needed)

### 2.2 Circuit & Trusted Setup
The circuit source is `buyer_nb500_tb5_m50.circom` (located at the
repository root).  The trusted setup has already been completed, producing:
```
zkp_nb500_tb5_m50/
├── buyer_nb500_tb5_m50_final.zkey    ← proving key
├── buyer_nb500_tb5_m50_js/           ← witness generator
├── proof.json                        ← a valid proof (for verification demo)
├── public.json                       ← public inputs of that proof
└── verification_key.json
```
To regenerate the setup (if needed), run:
```bash
cd /path/to/priME-deal
python src/10_Ours/RQ3/prime_deal_ZKP_bn254.py
```
This script also outputs a fresh `proof.json` and `public.json` for the
`n_b=500, t_b=5, m=50` configuration.

### 2.3 Measuring Proving & Verification Time
We measure the time of a single proof generation and verification using the
existing witness and trusted setup.

```bash
cd zkp_nb500_tb5_m50

# Generate witness (if not already present)
node buyer_nb500_tb5_m50_js/generate_witness.js \
  buyer_nb500_tb5_m50_js/buyer_nb500_tb5_m50.wasm \
  witness.json witness.wtns

# Prove
echo "=== Proving ==="
/usr/bin/time -f "Prove time (real): %e s" npx snarkjs groth16 prove \
  buyer_nb500_tb5_m50_final.zkey witness.wtns proof.json public.json

# Verify
echo "=== Verifying ==="
/usr/bin/time -f "Verify time (real): %e s" npx snarkjs groth16 verify \
  verification_key.json public.json proof.json

# Constraint count
npx snarkjs r1cs info buyer_nb500_tb5_m50.r1cs
```

### 2.4 Expected Output (representative)
```
=== Proving ===
Prove time (real): 2.12 s
=== Verifying ===
Verify time (real): 0.64 s
[INFO]  snarkJS: # of Constraints: 17500
[INFO]  snarkJS: # of Private Inputs:  ...
[INFO]  snarkJS: # of Public Inputs: 14
```
The prover time (≈ 2.1 s) and verifier time (≈ 0.64 s) correspond to the
values listed in the paper for this parameter set.  The circuit complexity
is dominated by `O(t_b · n_b)` field operations, consistent with the
asymptotic analysis.

## 3. Combining On‑chain and Off‑chain Results

For a representative transaction (`n_b=200`, `t_b=5`, `m=20`):
- Token reconstruction (off‑chain) : ≈ 0.5 s (see Buyer benchmark README)
- ZKP generation (off‑chain)       : ≈ 1.2 s (similar to the 500‑attr case)
- On‑chain `buy` gas              : ≈ 250 k gas
- Total buyer online latency       : < 2 s

These measurements confirm that PriME‑Deal achieves practical, interactive
performance while maintaining full bilateral attribute privacy.
